# wego Benchmark 快速上手

这份文档只回答 3 个问题：

1. 你现在要做什么
2. 每个任务怎么判断成功
3. 怎么把任务交给 agent 自动执行

## 你的当前目标

先用 `wego` 这个项目把评测流程跑通一次，而不是先追求复杂结论。

你要验证的是：

- agent 能不能完成任务
- 任务结果能不能被自动判定
- 任务能不能稳定复现
- `wego` 是否适合进入 benchmark 候选池

## 一共只做 4 个任务

### 任务 1：首次运行配置初始化

你要验证：

- 干净环境下运行 `wego`
- 是否会自动创建配置文件

看这里：`tasks/01-first-run-config.md`

### 任务 2：切换 frontend

你要验证：

- 同一份天气数据
- 换不同 frontend 后
- 输出格式是否真的变化

看这里：`tasks/02-frontend-switch.md`

### 任务 3：缓存创建与复用

你要验证：

- 第一次运行是否生成缓存
- 第二次相同参数运行是否复用缓存
- `--cache-ttl=0` 是否禁用缓存写入

看这里：`tasks/03-cache-reuse.md`

### 任务 4：离线 JSON 后端

你要验证：

- 不联网时是否仍能稳定运行
- 本地固定数据是否能支持 benchmark 复现

看这里：`tasks/04-offline-json-backend.md`

## 每个任务都要记录 4 类信息

### 1. 成功标准

必须写成可判断的条件，例如：

- 文件存在
- 输出可解析
- 退出码为 `0`
- 缓存文件时间未变化

不要写成：

- 看起来正常
- 应该没问题
- 输出像是对的

### 2. 可观测数据

对 `wego`，优先记录这些证据：

- `stdout`
- `stderr`
- `exit code`
- 文件是否存在
- 文件大小
- 文件修改时间

### 3. 专家轨迹

也就是你手工做测试时的真实步骤：

- 输入了什么命令
- 改了什么环境变量
- 观察到了什么
- 为什么判定成功或失败

### 4. agent 任务描述

agent 不要只收到“帮我测试一下”。

agent 需要看到：

- 目标是什么
- 前置条件是什么
- 允许做哪些操作
- 成功和失败怎么判定

## 最推荐的执行顺序

1. 先做任务 4，验证离线 JSON backend 能稳定运行
2. 再做任务 2，验证 frontend 切换
3. 再做任务 3，验证缓存
4. 最后做任务 1，验证首次运行配置初始化

这样做的原因是：

- 离线任务最稳定
- frontend 任务最容易定义 oracle
- 缓存任务需要你先熟悉参数和文件行为
- 首次运行配置任务最依赖隔离环境

## 你实际怎么填结果

先填：`results/manual-validation.md`

你手工跑完后，再填：`results/expert-traces.md`

等这些都稳定了，再用：`results/agent-prompts.md`

## agent 自动执行怎么开始

最简单的做法是：

1. 从一个任务卡中复制 `Agent Prompt`
2. 把 prompt 发给 agent
3. 让 agent 只完成一个任务
4. 用任务卡里的 oracle 检查结果

不要一开始让 agent 一次性完成全部 4 个任务。

## 当前建议

如果你今天只做一件事，就做下面这件：

- 先跑通 `任务 4：离线 JSON 后端`

因为这是最稳定、最不依赖外部环境、也最容易让 agent 自动化的任务。
