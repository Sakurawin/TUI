# 🎓 科研项目：基于 Agent-as-a-User 的生成式 TUI 可用性自动化评测
**研究目标：** 解决生成式 AI 开发的 TUI（文本用户界面）“能运行但不好用”的问题，通过模拟真实用户行为的 Agent 自动化评估其可用性（Usability）。

---

## 一、 输入信息复盘 (Input Analysis)

### 1. 核心痛点与动机
* **现状：** TUI 在开发者生态中极其重要，但其设计和测试高度依赖人工直觉。
* **问题：** 传统的自动化测试只能检测“功能是否正确”，无法检测“界面是否好用（UX/UX）”。
* **契机：** LLM（大语言模型）已具备操作 TUI 的能力（Computer-Use Agents），可以扮演“终端用户”来评估和指导 TUI 的生成与设计。

### 2. 技术路线与关键指标
* **方法论：** Agent-as-a-User（Agent 即用户）。
* **核心组件：**
    * `agent-tui`：底层驱动框架，负责按键操作、屏幕捕捉、系统调用监测（`strace`）。
    * **Oracle（真值验证）：** 基于任务成功信号、文件系统变更、专家轨迹对比。
* **度量指标 (Metrics)：**
    * **HCI 理论：** $K+T$ 指标（按键次数 + 思考时间）。
    * **认知负荷：** LLM Thinking Trace（Token 长度）。
    * **成功率：** 任务完成率（Success Rate）。

## 三、 第二阶段深挖：功能挖掘与基准库构建

这是你目前正准备进入的关键环节，重点在于**如何自动化提取测试题目**。

### 1. 挖掘算法 (Mining Algorithm)
* **原子功能 (Atomic)：** 通过静态代码扫描（搜寻 Keybinding 代码）或动态探针（尝试所有按键）提取单一操作。
* **复合功能 (Composite)：** 结合 LLM 对 `README` 的理解，模拟用户完成复杂任务（如：搜索 -> 选中 -> 删除）。

### 2. 成功信号定义 (Oracle)
* **状态 Oracle：** 任务执行后，检查 Docker 容器内的文件、进程、网络状态是否发生了预期变化。
* **视觉 Oracle：** 截图对比，检查 TUI 屏幕上是否出现了特定关键词或 UI 布局。
* **轨迹 Oracle：** 将 Agent 的操作路径与“专家操作路径（Golden Trace）”进行相似度对比。

## 四、 当前执行：第一步做什么？

当你已经下载了一个 `awesome-tui` 项目到 VS Code 后，应按照以下步骤启动：

1.  **环境隔离 (Dockerize)：**
    * 为该项目编写 `Dockerfile`。
    * 在镜像中安装 `strace`、`gdb` 等监控工具以及项目运行所需的编译器/运行环境。
2.  **静态摸底 (Code Analysis)：**
    * 在 VS Code 中通过全局搜索（如 `key`, `input`, `bind`）确定该项目的交互逻辑。
    * 手动运行项目，记录几个典型操作的按键流。
3.  **定义测试用例 (Task Schema)：**
    * 创建一个 JSON 文件，描述一个具体的任务。
    * *示例：* `{ "task": "删除名为 test.txt 的文件", "oracle_check": "ls test.txt 返回空" }`。
4.  **录制专家轨迹：**
    * 手动完成上述任务，保存操作序列作为后续评测 Agent 效率的基准。
